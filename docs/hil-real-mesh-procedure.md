# Single-board real-mesh HIL procedure (MeshCadet M1)

Run MeshCadet on **ONE** T-Deck Plus and interop with the operator's existing
**STOCK** MeshCore nodes + companion apps **without reflashing them**. Bootstrap
is manual contact-add: the operator registers MeshCadet's public key in the app.
No advert is ever emitted (charter purity).

This exercises the DM→ACK / ACK-recv / GRP_TXT-recv paths fed with **real**
identities sourced from a gitignored local config.

> Scope: stops at the flash boundary. The operator owns flashing + HIL. **No HIL
> pass is asserted by this procedure** — these are the steps to run it.

## 1. Populate the gitignored key config

The `hil` build reads real keys from `firmware/src/hil_keys.rs`, which is
**git-ignored** (never committed). Start from the committed placeholder:

```sh
cp firmware/src/hil_keys.example.rs firmware/src/hil_keys.rs
```

Edit `firmware/src/hil_keys.rs` and set:

| Const | Value |
|-------|-------|
| `HIL_SELF_SEED` | A random 32-byte Ed25519 seed for MeshCadet. Pin it once — it makes MeshCadet's pubkey stable across reflash + NVS-erase, so you register the contact only once. |
| `HIL_PEER_PUBKEY` | The operator's peer node Ed25519 **public** key (32 bytes), copied from the companion app contact/QR export. |
| `HIL_CHANNEL_SECRET` | The real group-channel secret. 128-bit channel → put the 16-byte secret in the first 16 bytes; 256-bit → fill all 32. |
| `HIL_CHANNEL_KEY_LEN` | `16` for a 128-bit channel (hash = `SHA256(secret[0:16])[0]`), `32` for a 256-bit channel (hash = `SHA256(secret)[0]`). Must match the real node exactly. |

**Secret hygiene:** `hil_keys.rs` holds secret material (seed, channel secret).
Never commit it, never paste its values into logs, chat, or notes. The firmware
prints only public, on-the-wire bytes (MeshCadet's own pubkey, the 1-byte channel
hash, 1-byte routing hashes) — never the seed or channel secret.

## 2. Build + flash

**Prerequisites:** `espflash` installed (`cargo install espflash`) and the
T-Deck Plus connected via USB.

**Flash + open serial monitor** — `.cargo/config.toml` sets
`runner = "espflash flash --no-skip --monitor"`, so `cargo run` builds, flashes,
and attaches the monitor in one step (`--no-skip` forces the freshly-built app
to be written every run, so incremental edits land without a `cargo clean`):

```sh
cd firmware
cargo run --features hil
```

If `espflash` cannot auto-detect the port, pass it explicitly after `--`:

```sh
cargo run --features hil -- --port /dev/ttyACM0
```

**Confirm the flash landed the latest build.** At boot the firmware logs an
authoritative build line:

```
firmware build: <short SHA[-dirty]>
```

Compare it against `git rev-parse --short HEAD` in `firmware/` (append
`-dirty` if `git diff --quiet HEAD` reports changes); they must match. Use
THIS line — not the esp-idf `App version:` boot tag — as the
"did my `cargo run` land?" check. The esp-idf tag is generated by esp-idf-sys's
CMake and is not re-emitted on an incremental `cargo run`, so it can freeze at
an older `git describe` (e.g. show a stale `-dirty` hash) even though the Rust
app relinked fresh (a prior fresh-flash-confirmation bug fixed this gap).

For an optimized image, add `--release`:

```sh
cargo run --release --features hil
```

**Build only** (no flash — useful to verify compilation without a connected board):

```sh
cargo build --features hil --target xtensa-esp32s3-espidf
```

## 3. Register MeshCadet's pubkey in the companion app

Watch MeshCadet's serial at boot:

```
identity ready: pub_hash=0x<hh>, pubkey=<64 hex chars>
peer pub_hash=0x<hh>
channel hash=0x<hh>
```

1. Copy the full 64-hex-char `pubkey` line.
2. In the companion app, **manually add a contact** using that public key (the app
   may call this "add contact by key" / "import key"). Do **not** wait for an
   advert — MeshCadet emits none.
3. Confirm the contact's short hash in the app matches the printed `pub_hash`.
4. Confirm the printed `channel hash` matches the app's channel hash for the
   channel whose secret you configured.

Both ends now know each other's pubkey, so ECDH works in both directions.

## 4. DM + ACK, both directions

**MeshCadet → peer (and ACK back):** MeshCadet auto-sends a periodic TEST DM to
`HIL_PEER_PUBKEY` every 30 s. Expect:
- App: the DM arrives from MeshCadet's contact.
- Serial: `TX: N bytes ...` then, when the node replies, `ACK received: matches
  last-sent DM (ack_hash=...)`.

**peer → MeshCadet (and ACK back):** send a DM from the app to MeshCadet's
contact. Expect:
- Serial: `RX DM from 0x<hh> ts=... "<text>"` then `TX ACK queued for 0x<hh>`.
- App: delivery/ACK indicator for that message.

## 5. Channel GRP_TXT receive

Post a message to the configured channel from the app. Expect on serial:

```
RX GRP_TXT (channel 0x<hh>) ts=... "<sender>: <message>"
```

If nothing decodes, the most common cause is a channel-hash convention mismatch:
re-check `HIL_CHANNEL_KEY_LEN` (16 vs 32) and that the secret bytes match.

## 6. Pull-only telemetry / location (acceptance predicate #4)

Verifies the enabled-contact-answered / non-enabled-dropped contract. Two
checkpoints must both hold, and the request path is gated **at boot** (see note).

> **Two request paths.** The stock MeshCore companion app's telemetry/location
> **button** does NOT send a `?loc` text DM — it sends a native
> `PAYLOAD_TYPE_REQ` (0x00) with `REQ_TYPE_GET_TELEMETRY_DATA` (0x03) and waits
> for a `PAYLOAD_TYPE_RESPONSE` (0x01). This is the path that predicate #4
> exercises. MeshCadet also answers a bespoke `?loc` text DM (a fallback for
> text-only clients); it is NOT what the companion's telemetry UI uses.
> Historically the firmware answered ONLY `?loc`, so the companion button always
> showed "Telemetry unavailable…" while every host test (which only exercised
> `?loc`) stayed green — a HIL-only regression host tests couldn't catch.

### Enabled contact is answered (native companion request — predicate #4)

1. Provision the companion as a contact with telemetry enabled
   (`meshcadet add-contact --pubkey <HEX64> --telemetry`).
   **Reboot the device** so the dispatcher's allowlist + telemetry gate (a boot
   snapshot of the provisioned config) pick up the change — the CLI prints this
   reminder. `list-contacts` showing `telemetry=true` is necessary but **not**
   sufficient on its own: the gate is read at boot.
2. In the companion app, open MeshCadet's contact and tap the **telemetry /
   request location** action (the same button that previously showed
   "Telemetry unavailable…").
3. **Expected serial** (the wire-first checkpoints for the native path):
   - `RX REQ telemetry pull from 0x<hh> (tag=0x........): telemetry_enabled=true (fix=...)`
     — the native REQ was received, decrypted, and the gate passed.
   - `TX telemetry RESPONSE to 0x<hh> (tag=0x........): location` (or
     `no-fix (tag only)` if GPS has no fix yet).
4. **Expected on the companion:** the telemetry/location view populates (GPS
   position when a fix exists) instead of the "Telemetry unavailable…" error.
   The reflected `tag` is what lets the app match the response to its request.

### Enabled contact is answered (bespoke `?loc` fallback)

1. Provision as above.
2. From the companion, send a DM with the text `?loc` (case-insensitive; leading
   whitespace is tolerated).
3. **Expected serial** (the three wire-first checkpoints):
   - `RX DM from 0x<hh> ts=... "?loc"` — request received + decrypted.
   - `RX DM telemetry pull detected from 0x<hh>: telemetry_enabled=true (fix=...)`
     — recognized as a pull and the gate passed.
   - `TX telemetry reply to 0x<hh>: location (age=<N>s)` (or `loc:nofix` if GPS
     has no fix yet).
4. **Expected on the companion:** a DM back from MeshCadet of the form
   `loc:lat=XX.XXXXXXX,lon=YY.YYYYYYY,age=Zs` (or `loc:nofix`). The `age` field is
   the required visible fix-age/timestamp.

### Non-enabled contact is dropped

1. Provision (or re-provision) the contact with telemetry **disabled** (omit
   `--telemetry`); reboot.
2. Trigger a telemetry pull (companion button and/or `?loc`) from that contact.
3. **Expected serial:** `telemetry_enabled=false` on the matching path
   (`RX REQ telemetry pull ... telemetry_enabled=false` for the native button, or
   `RX DM telemetry pull detected ... telemetry_enabled=false` for `?loc`) then a
   suppression diagnostic — **no** `TX telemetry RESPONSE` / `TX telemetry reply`.
   A `?loc` DM is still ACKed (the contact is in the allowlist); a native REQ is
   not ACKed (MeshCore answers a REQ only with a RESPONSE). Either way the
   companion sees no telemetry.

### If the enabled contact is NOT answered

Read the serial to localize, top to bottom:
- No `RX REQ telemetry pull` / `telemetry pull detected` line → the request never
  reached the handler. For the native path, confirm the companion actually sent a
  `PAYLOAD_TYPE_REQ` (some app versions gate telemetry behind path discovery);
  for `?loc`, confirm the exact DM text on the wire.
- `telemetry_enabled=false` while `list-contacts` shows `true` → the boot
  snapshot is stale; **reboot** (allowlist + gate load at boot). If it persists
  after reboot, suspect a stale duplicate contact entry.
- `TX telemetry RESPONSE`/`TX telemetry reply` present but companion shows
  nothing → reply TX/path issue, not detection/gate.

## Out of scope (separate M1.5)

Cold-path path-learning — needs a repeater in-path plus a PATH-return emit the
firmware does not yet have. DM/ACK here use flood (0-hop) routing.

## Troubleshooting: "TX works, RX dead"

Symptom: serial logs `radio: SX1262 configured` and a periodic `TX: N bytes,
Mms airtime`, but **zero** RX lines over minutes — no `RX DM`, no `ACK
received`, no `RX GRP_TXT`, not even an `RX: CRC error`. Work this decision tree
top to bottom; each step is cheaper to check than the one below it.

1. **Preset match.** Confirm the peer is on the EXACT MeshCadet preset:
   910.525 MHz / 62.5 kHz / SF7 / CR4/5. A bandwidth or SF mismatch is
   demodulated as noise — no RxDone. (Earlier bug: BW code `0x04` = 125 kHz
   instead of `0x03` = 62.5 kHz.)

2. **Peer firmware version / ACK format.** A v1.16 peer emits a 6-byte ACK;
   MeshCadet's `compute_ack_hash` matches v1.15's 4-byte ACK. Version skew shows
   up as DMs arriving but ACKs never matching — not total RX silence.

3. **LoRa sync word.** This is the classic *total-silence* cause. The SX1262
   matches the LoRa sync word **in hardware during demod**; a mismatch drops the
   packet at the PHY before RxDone or CrcErr can fire — hence "not even a CRC
   error". Stock MeshCore v1.15 nodes use RadioLib's **PRIVATE** word
   (`RADIOLIB_SX126X_SYNC_WORD_PRIVATE` = `0x12`), which RadioLib expands to
   register bytes `{0x14, 0x24}` (= `0x1424`) at `REG_LORA_SYNC_WORD`
   (0x0740/0x0741). The PUBLIC word `0x34` expands to `0x3444`. MeshCadet must
   write `0x1424` (see `radio.rs` `SYNC_WORD_HI`/`SYNC_WORD_LO`). Writing the
   public `0x3444` makes the radio deaf to every stock node.

4. **DIO1 / RX-mode.** Confirm `try_receive` (`radio.rs`): RxDone+CrcErr IRQs are
   routed to DIO1 (mask `0x0042`), DIO1 is wired to GPIO45, and the SetRx
   hardware timeout actually covers the poll window. The SetRx `Timeout` field is
   a 24-bit count in **15.625 µs steps** (steps = ms × 64); an off-by-1000 in that
   conversion (dividing by 15625 instead of 15.625) sets a ~47 µs window for a
   50 ms request, dropping the radio out of RX almost immediately — deaf in
   practice even with the correct sync word. Also confirm the dispatcher returns
   the radio to RX after each TxDone rather than leaving it in standby.

## Touch UI HIL acceptance procedure

> These steps must be run against a **production build** (no `hil` feature flag)
> on a T-Deck Plus that has been provisioned via the admin CLI.
> They verify the touch UI's emoji-rendering acceptance criteria.

### Prerequisites

1. T-Deck Plus provisioned with at least one contact and one channel.
2. Firmware flashed with the production image (no test feature flags).
3. operator's MeshCore node active on air (same channel, DM contact registered).

### A. First-boot / Unprovisioned screen

1. Wipe NVS: `idf.py erase_flash` OR `esptool.py erase_region <nvs offset> <nvs size>`.
2. Boot the device.
3. **Expected:** "📻 MeshCadet" wordmark + pulsing glow + "Ask an admin to connect via USB" instruction + device public key hex displayed.
4. **Expected:** No radio frames transmitted (radio never initialises unprovisioned).
5. Connect CLI, provision, device reboots to contact list.

### B. Contact list + channel tabs

1. Provision and reboot.
2. **Expected:** Contact list screen visible; at least the provisioned contact appears.
3. Tap "📡 Channels" tab — channel(s) should appear.
4. Tap back to "📬 Messages" tab.

### C. Message view + incoming DM render

1. From the operator's node, send a DM to MeshCadet.
2. **Expected (visual notification):** Screen border flashes green for ~300 ms.
3. **Expected:** Unread badge increments on the contact row.
4. Tap the contact row — **Expected:** Message view opens; incoming message text renders correctly including any emoji codepoints.
5. Tap "✏ Write" — compose screen opens.

### C.1 Chat autoscroll (regression check)

Regression check for the message-view scroll defect: the list must pin to the
newest message on view entry and on every new-message arrival, without a
manual scroll gesture.

1. From the operator's node, send MeshCadet enough DMs (8–10 short messages)
   to overflow the 320×240 message-view viewport.
2. Navigate away from Message view (back to the contact list), then tap the
   contact row again. **Expected:** Message view opens already scrolled to
   the newest (bottom-most) message — no manual scroll needed to see it.
3. With Message view still open, have the operator's node send one more DM.
   **Expected:** the new message appears already in view at the bottom the
   moment it arrives — the view does not require a manual scroll down, and
   it does not stay pinned to wherever the user had scrolled to previously.
4. Repeat step 3 twice more in a row (rapid back-to-back DMs). **Expected:**
   each arrival re-pins to the new bottom — behavior is consistent across
   repeats, not intermittent.

### D. Emoji entry + compose + send

1. On compose screen, tap the 😀 button to open the emoji picker grid.
2. **Expected:** 40-cell emoji grid renders (5 columns × 8 rows), all cells tappable.
3. Tap any emoji — **Expected:** emoji codepoint inserted into the draft text area; picker closes.
4. Alternatively: type `:smile:` via the physical keyboard — **Expected:** shortcode appears in the text area (NOT expanded yet; expansion happens on send).
5. Tap "📤 Send" — **Expected:** DM frame transmitted; ✓ indicator appears; on ACK receipt ✓ changes to ✓✓.

### E. PIN entry surface

1. Navigate to the admin menu (tap the PIN entry route).
2. **Expected:** PIN entry screen shows 4 dots + numpad.
3. Enter 3 digits — **Expected:** first 3 dots fill; confirm button remains disabled.
4. Enter 4th digit — **Expected:** confirm button enables.
5. Tap ⌫ — **Expected:** dot empties.
6. Enter correct PIN, tap ✓ — **Expected:** `PinSuccess` notification fires (green flash + two-tone chirp).
7. Enter wrong PIN — **Expected:** `PinError` notification fires (red flash + descending tone).

### F. Notification model

1. Enable audible notifications (default is on).
2. Receive a DM — **Expected:** buzzer plays two-tone chirp (880 Hz / 120 ms then 1320 Hz / 80 ms).
3. Receive a group message — **Expected:** different tone pattern (660 Hz / 100 ms then 880 Hz / 100 ms).
4. Disable audible notifications for `IncomingDm` via the admin settings menu.
5. Receive another DM — **Expected:** visual flash only; no buzzer.

### G. Conversation history rotate + export freshness (acceptance predicate #5, history half)

Regression check for a previously-fixed `export-history` staleness defect:
a fresh export must reflect messages that
arrived *after* a prior export, for both DMs and channel group text — not
just whatever was in history the first time the CLI was run.

1. Run `meshcadet export-history` once. Note the entry count (may be 0 on a
   fresh device).
2. From the operator's node, send MeshCadet a DM.
3. Post a message to the shared channel from the operator's node.
4. Re-run `meshcadet export-history`. **Expected:** both the new DM (`DM`
   type) and the new channel message (`GRP` type) appear as the newest two
   entries (oldest-first ordering — they are last), with correct `timestamp`
   and `text`.
5. Repeat steps 2–4 once more. **Expected:** the second round's DM and GRP
   messages also appear on the next export — freshness holds across multiple
   rounds, not just the first re-run.
6. Send `MAX_HISTORY_ENTRIES + 1` (33) messages of either type in total across
   this and prior steps, then export. **Expected:** exactly 32 entries,
   oldest-first, with the very oldest entry evicted (matches the ring-buffer
   rotation-at-bound unit test in `protocol/src/history.rs`).

If the GRP entries are silently absent while DM entries appear fresh, suspect
`handle_grp_txt` (`firmware/src/main.rs`) not appending to `HISTORY` — this was
exactly the root cause of the original defect (channel receipt rendered to the
on-device UI but was never persisted to the exportable history store).

### H. Screen sleep (backlight-off) + wake-swallow

Regression check for the global first-input-interceptor: waking the screen
must NEVER itself change app state (no navigation, no button activation, no
text input) — the wake-triggering input is consumed to wake only. Incoming
messages must NOT wake the screen.

1. From the admin menu, set "💤 Screen sleep" down to `10s` (tap "−" from the
   default 30s). Tap back to the contact list.
2. Leave the device untouched for 10+ seconds. **Expected:** the backlight
   turns off (panel goes dark); the device does not reboot or hang — a key
   press or tap still wakes it (next steps).
3. Tap anywhere on the dark screen. **Expected:** backlight comes back on
   immediately, showing the contact list exactly as it was — the tap does
   NOT open a conversation, even if the tap landed on a contact row.
4. Repeat steps 2–3 but let it sleep while the **PIN entry** screen is shown
   (tap the settings gear, then let it sleep). Tap one of the numpad digit
   keys to wake it. **Expected:** screen wakes showing the PIN dots exactly as
   left (still 0 digits entered) — the waking tap must NOT register as a
   digit press. This is the sharpest case: confirm the entered-digit dot
   count is unchanged by the wake tap.
5. Let the screen sleep again from the contact list. Wake it with the
   **physical keyboard** instead of touch (press any key). **Expected:** same
   swallow behavior — screen wakes, no key reaches the focused widget (no
   character appears if a text field happened to have focus).
6. From the operator's node, send MeshCadet a DM while the screen is asleep.
   **Expected:** the screen stays asleep — an incoming message does not wake
   it (explicit product decision; the message is waiting on the next manual
   wake).
7. Set "💤 Screen sleep" to `0` via "−" (holding at the floor). **Expected:**
   the display reads "Never". Leave the device untouched well past any
   previous timeout. **Expected:** the screen never sleeps.
8. Set "💤 Screen sleep" back to a nonzero value (e.g. `30s`), then power-cycle
   the device. Re-open the admin menu. **Expected:** the value shown matches
   what was set before reboot (persisted in the `mc_rts` NVS namespace,
   separate from the provisioning config blob).

### I. Keyboard backlight follows screen

Regression check that the physical keyboard's own backlight LED (driven by the
ESP32-C3 keyboard co-processor, I2C `0x55` — separate from the display's
GPIO42 backlight) is 1:1 coupled to the same wake/sleep transitions as §H,
with no independent timer.

1. Power on the device (or power-cycle it). **Expected:** the keyboard
   backlight is lit by the time the first screen (wordmark or contact list)
   is visible — same as the display backlight, at boot.
2. From the admin menu, set "💤 Screen sleep" down to `10s`, then return to
   the contact list and leave the device untouched for 10+ seconds.
   **Expected:** the keyboard backlight turns off in the same moment the
   display backlight does (both dark together, not one lagging the other).
3. Tap anywhere on the dark screen. **Expected:** the keyboard backlight
   relights in the same moment the display backlight does.
4. Repeat step 2, then wake with the **physical keyboard** instead of touch
   (press any key, per §H step 5). **Expected:** the keyboard backlight
   relights along with the display — a key press both wakes the screen AND
   relights the very backlight it was pressed under.
5. Throughout steps 1–4, confirm keypresses and touches are still read
   correctly (type a few characters into Compose, tap a contact row) — the
   added I2C backlight writes must not disrupt keypress/touch reads sharing
   the I2C1 bus.

If the keyboard backlight does not track the display (stays off at boot, or
one lags/never follows the other), suspect `KeyboardDriver::set_backlight`
not being called from `UiRuntime::new`/`wake_screen`/`sleep_screen`
(`firmware/src/ui/mod.rs`), or the co-processor's I2C command byte
(`CMD_SET_BACKLIGHT` in `firmware/src/ui/keyboard.rs`) not matching the
flashed co-processor firmware's `LILYGO_KB_BRIGHTNESS_CMD`.

### J. Incoming-message visual: keyboard-backlight blink while asleep

Regression check for the redefined incoming-message VISUAL notification: no
screen-border flash for `IncomingDm`/`IncomingGroupMsg` at all (awake or
asleep); asleep, a keyboard-backlight blink loop takes its place and must not
fight with §I's screen-follows rule over the same backlight.

1. With the screen **awake** and visual notifications enabled, have the
   operator's node send MeshCadet a DM. **Expected:** no screen-border flash,
   no keyboard-backlight blink — nothing visual happens (audible tone, if
   enabled, still plays — unaffected by this change).
2. Let the screen sleep (§H). Send a DM. **Expected:** the keyboard backlight
   blinks ~3 times immediately, then goes dark (matching §I's asleep-off
   state) until the next burst — roughly every 5 seconds — repeating
   indefinitely.
3. While the loop is still running, send a second DM (and/or a channel
   message). **Expected:** still a single blink loop on the original ~5s
   schedule — no faster blinking, no second overlapping loop, no restart of
   the burst timing.
4. Wake the device (tap or key press, per §H). **Expected:** the blink loop
   stops immediately and the keyboard backlight comes on solid with the
   display — same moment, no lingering blink tail.
5. Let the screen sleep again with no pending message. **Expected:** the
   keyboard backlight simply stays off (§I's steady-state) — confirm the
   blink loop does not fire spontaneously without an incoming message.
6. Disable visual notifications for `IncomingDm` in the admin menu, then
   repeat step 2. **Expected:** no blink at all while asleep (visual pref
   gates the blink the same way it gated the old flash) — the keyboard
   backlight stays off; audible tone (if enabled) is unaffected.

If the blink races with §I's screen-follows rule (backlight flickers
erratically, or relights immediately after a blink instead of returning to
off), suspect two writers on the keyboard backlight instead of the single
`sync_keyboard_backlight` arbiter (`firmware/src/ui/mod.rs`) — `wake_screen`/
`sleep_screen` must not call `KeyboardDriver::set_backlight` directly.

### K. Accent colour order

Regression check for the ST7789 colour-order fix: the panel wires its
subpixels BGR, but the driver was telling it RGB, so every accent that should
render `#00b4ff` blue rendered `#ffb400` orange instead (R and B channels
swapped for every colour, not just blue).

1. Boot to the contact list. **Expected:** the selected-row highlight and its
   icon accent render **blue** (`#00b4ff`), not orange.
2. Open a DM thread with at least one sent, acked message. **Expected:** the
   message-bubble border renders blue, and the acked message's `✓` renders
   blue — not orange, not red/pink. (Before the ack-indicator fix in §L,
   this was a double-check `✓✓`; the ack indicator is now always a
   single check that only changes color — see §L.)
3. Open Compose. **Expected:** the send button (when a draft is present) and
   the emoji-picker toggle (when open) render blue.
4. While on any screen with a visible accent, also confirm the things that
   are NOT supposed to be blue still read correctly: black backgrounds stay
   black, white/light body text stays white/grey (`#607080`-style muted
   text), and — if a screen-border flash or PIN-dot fill is visible — reds
   and greens used elsewhere render as red/green, not swapped or inverted.
   This step exists because a naive fix (e.g. hand-swapping R/B on one accent
   constant) would "fix" blue by making every other colour wrong instead of
   correcting the actual root cause (the controller's subpixel-order bit).

**Superseded by §N:** the original diagnosis above (panel wired BGR, driver
told RGB) turned out to be confounded by the missing color-inversion fix that
landed later (§M) — see §N for the corrected mechanism and the current
correct setting (`ColorOrder::Rgb`). If any accent still renders orange or
yellow/amber, suspect `ColorOrder` set to `Bgr` in
`firmware/src/ui/display.rs` (the CURRENT correct value is `Rgb`, not `Bgr` —
this inverts the original suspect-note below). If accents are now correct but
some OTHER colour (red/green) looks swapped or inverted, suspect a second,
compensating channel-swap added elsewhere in the pixel-push path
(`flush_line_range`, `draw_flash_border`) stacking with the `ColorOrder` fix
— there should be exactly one point of truth for subpixel order (the
`color_order(...)` call in `display.rs`), not two.

### L. Ack indicator is a single check that flips color

Regression check for the ack-glyph simplification: the outbound ack indicator
is now ALWAYS a single check (`✓`) — there is no double-check (`✓✓`) glyph
anywhere in the UI. The only ack signal is color.

1. Send a DM (or a channel/group message) and immediately look at the
   just-sent bubble's indicator. **Expected:** a single `✓`, rendered grey
   (`#607080`) — not two checks, not blue yet.
2. Once the peer's ACK/repeat is received (§4/§5, or §C for a live DM ack),
   look at the same message's indicator again. **Expected:** the glyph is
   STILL a single `✓` — it must NOT become `✓✓` — but its color has flipped
   to accent blue (`#00b4ff`, per §K's color-order fix).
3. Repeat step 1–2 for a channel/group outbound message (§5). **Expected:**
   identical single-check, grey→blue behavior — DM and channel/group outbound
   messages drive the same `acked` bit and render identically.
4. Scroll back through history with a mix of acked and unacked outbound

### Q. Battery gauge doesn't inflate to 100% while charging

On-device check for the charge-inflation bug: gauge previously read a false
~100% + "not charging" while on external power, then jumped back to 100%/
charging on a second plug-in, and collapsed to the true ~36% only once
unplugged (see `battery.rs` module docs' "Fix" section for the mechanism —
percent is held at the last known good off-power voltage while on power).
**Note:** the original fix latched charging from a voltage-*rise* trigger;
that trigger was replaced by a reliable impossible-voltage threshold check
(see §R below) after a HIL capture
showed the rise trigger failing to engage in a case a rise can't see. The
percent-freeze *behavior* this section checks is unchanged; only the
mechanism deciding *when* to freeze changed.

Requires a pack at a **known-partial** charge (not freshly topped off) —
e.g. run the device on battery until the admin-menu battery row reads
somewhere in the 20–70% band before starting this check.

1. On battery only (unplugged), open the admin-menu battery row (or run
   `meshcadet --port <PORT> status`). Note the reported percent — call it `P0`
   — and confirm `charging: false`.
2. Plug in external power (USB-C). **Expected:** within one poll interval
   (~2 s) the status flips to `charging: true`, and the reported percent
   stays at (or within a couple of points of) `P0` — it must NOT jump toward
   ~100% just because the pack's on-charger terminal voltage is elevated.
3. Leave it charging for several minutes, long enough for the pack to reach
   the charger's float/CV plateau (voltage stops climbing). **Expected:**
   `charging` stays `true` throughout — it must not flip back to `false` once
   the voltage plateaus (the exact failure mode from the original HIL
   report), and the reported percent still does not jump to 100%.
4. Unplug external power. **Expected:** within one poll interval, `charging`
   flips to `false`, and the reported percent settles at a value consistent
   with real charge gained since step 1 (some increase off `P0` is normal and
   expected — that's real, physical charging progress) — not a large
   discontinuity toward/away-from 100% (i.e. no "100%-then-collapse" jump).
5. Repeat one plug/unplug cycle. **Expected:** `charging` and percent track
   the same way each time — no first-plug/second-plug inconsistency (the
   other half of the original HIL report).

If step 2 or 3 shows percent jumping toward 100%, suspect `settled_mv` isn't
actually being frozen — check `battery_poll_step`'s power branch. If step 3
shows `charging` flipping back to `false` on the plateau, suspect
`EXTERNAL_POWER_MV_THRESHOLD` regressed to something the pack's genuine
on-charger voltage can reach (it must stay well above [`BATTERY_FULL_MV`]).
If step 4 never clears `charging`, the raw reading post-unplug may not have
settled back under the threshold yet — wait one more poll interval before
concluding a regression.

### R. Plugged-in gauge holds last unplugged SoC instead of a false 100% off the charge rail

Regression/acceptance check for the follow-on HIL capture that superseded §Q's
rise-based trigger: with the ADC calibration fix landed, a plugged-in pack's raw
reading was captured at **4888 mV** — physically impossible for a
single-cell pack (max ~4200 mV full) — proving the ADC divider reads the
USB/charge rail directly whenever external power is connected, not just an
IR-drop-elevated battery terminal. `battery.rs` now treats any raw reading
above [`EXTERNAL_POWER_MV_THRESHOLD`] (4300 mV) as a reliable, stateless
external-power-present signal, replacing the voltage-rise trigger that failed
to engage in this exact case (see `battery.rs` module docs' "Fix" section).

Also covers the boot-on-power edge: a device power-cycled while still plugged
in must report `charging: true` from its very first status read, not just
"not charging" until the next poll (the residual gap the rise trigger could
never close — see module docs' "Fix" section's boot-case caveat).

1. On battery only (unplugged), run `meshcadet --port <PORT> status`. Note the
   reported `battery` percent (call it `P0`) and `battery raw` mV (call it
   `V0`) — confirm `charging: false` and `V0` is a plausible pack voltage
   (well under 4300 mV).
2. Plug in external power (USB-C). **Expected:** within one poll interval
   (~2 s), `battery raw` jumps to a reading **above 4300 mV** (the
   charge-rail read, not the battery), `battery` percent stays at `P0` (no
   jump toward 100%), and the status reports charging/on-power.
3. Leave it on power for several minutes. **Expected:** `battery raw` stays
   above 4300 mV (may drift, e.g. LED going from charging to charge-complete)
   and `battery` percent stays pinned at `P0` throughout — no discontinuity,
   no reversion to a rise-based read.
4. Unplug external power. **Expected:** within one poll interval, `battery
   raw` drops back under 4300 mV to a plausible pack voltage, `charging`
   clears, and `battery` percent settles at a value consistent with real
   charge gained since step 1 (some increase off `P0` is normal) — not a
   large discontinuity toward/away-from 100%.
5. Repeat one plug/unplug cycle. **Expected:** the same behavior both times —
   no dependency on a prior-poll baseline (the failure mode that made the old
   rise trigger miss this case; a threshold check needs no such history).
6. Power off the device, then power it back on **while still plugged into
   external power** (boot-on-power edge). **Expected:** the very first
   `status` read after boot already reports charging/on-power — it must not
   take a second poll, an unplug, or a replug to see `charging: true`.

If step 2 shows `battery raw` above 4300 mV but `battery` percent still jumps
toward 100%, suspect the percent basis isn't reading `settled_mv` (i.e.
`BatteryStatus::percent` regressed to deriving from live/raw mV again — see
`BatteryDriver::status`). If `battery raw` never exceeds 4300 mV while
plugged in on this specific hardware sample, the on-power rail voltage may
differ from the 4888 mV this threshold was calibrated against —
recapture the actual raw mV and adjust `EXTERNAL_POWER_MV_THRESHOLD` to stay
well above [`BATTERY_FULL_MV`] but below the observed on-power reading. If
step 6 fails to report charging on the first read, check that
`BatteryDriver::new` routes its initial sample through `battery_poll_step`
rather than seeding `cached_charging: false` unconditionally.
